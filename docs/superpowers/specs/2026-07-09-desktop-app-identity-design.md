# Desktop App Identity — Design

**Date:** 2026-07-09
**Status:** Approved

## Goal

Give crew a real application identity on all three desktop platforms: it should
appear in macOS Spotlight/Launchpad, the Windows Start menu, and Linux
application menus, each with its own icon — while preserving the existing
single-binary distribution (`~/.local/bin/crew`) and the `/update` flow.

## Decisions

- **Platforms:** macOS, Windows, and Linux in this pass.
- **Icon:** original artwork designed in-repo as an SVG; platform formats
  generated by a script and committed.
- **Mechanism:** the crew binary self-registers (writes the bundle / shortcut /
  desktop entry itself); no CI-built installers, no changes to the
  single-binary release archives.
- **Trigger:** automatic on GUI launch (idempotent, staleness-checked, run off
  the winit thread), plus an explicit `crew install-app` subcommand which
  `install.sh` also invokes. Opt-out via `CREW_NO_APP_INSTALL=1` (or config).

## 1. Icon assets

- `assets/icon/crew.svg` — master artwork. A crew/hive-flavored mark designed
  to stay legible from 16 px to 1024 px.
- `scripts/gen-icons.sh` — run on a dev machine (requires an SVG rasterizer
  such as `rsvg-convert`, and `iconutil` on macOS). Produces the committed
  derivatives:
  - `assets/icon/crew.icns` — macOS iconset (16–1024 px, including @2x).
  - `assets/icon/crew.ico` — Windows multi-size (16, 24, 32, 48, 64, 128, 256).
  - `assets/icon/crew-{32,128,256,512}.png` — Linux hicolor sizes; the 512 px
    PNG also feeds the macOS runtime Dock icon.
- Generated files are **committed** so CI and the binary never need SVG
  tooling; the binary embeds them with `include_bytes!`.

## 2. Windows exe icon (build time)

`crates/crew-app/build.rs` uses the `winresource` crate (active only when
building for Windows) to embed `crew.ico` as the exe's icon resource. This
gives Explorer, the taskbar, and any shortcut the correct icon with no runtime
work.

## 3. Self-registration

New module `crates/crew-app/src/appregister.rs` with one implementation per
OS behind `cfg`, all idempotent.

### macOS

Writes `~/Applications/Crew.app/`:

```
Crew.app/Contents/
├── Info.plist            CFBundleName=Crew, CFBundleDisplayName=Crew,
│                         CFBundleIdentifier=io.github.ashishtyagi10.crew,
│                         CFBundleExecutable=crew, CFBundleIconFile=crew,
│                         CFBundleShortVersionString=<crate version>,
│                         LSMinimumSystemVersion, NSHighResolutionCapable
├── MacOS/crew            symlink → the real binary
└── Resources/crew.icns   embedded bytes
```

- Symlink target rule: if `~/.local/bin/crew` exists, point at it (the stable
  installed location); otherwise fall back to `std::env::current_exe()`. This
  stops auto-registration from a dev build repointing the bundle at a
  `target/` path that later disappears.
- Symlink (not copy) means `/update` replacing `~/.local/bin/crew` never
  leaves a stale app copy.
- Spotlight and Launchpad index `~/Applications` automatically; no
  `lsregister` call needed (may be added later if indexing proves slow).
- No code signing: the bundle is created locally and never quarantined, so
  Gatekeeper does not block it.

### Linux

- `~/.local/share/applications/crew.desktop`:
  `Type=Application`, `Name=Crew`, `Exec=<absolute binary path>`,
  `Icon=crew`, `Terminal=false`, `Categories=Development;Utility;`,
  `StartupWMClass=crew`.
- Icon PNGs into `~/.local/share/icons/hicolor/<size>x<size>/apps/crew.png`.
- Best-effort `update-desktop-database`/`gtk-update-icon-cache` if present;
  ignore failures.

### Windows

- `%APPDATA%\Microsoft\Windows\Start Menu\Programs\Crew.lnk` created with the
  `mslnk` crate (pure Rust .lnk writer, no COM). Target = absolute exe path;
  icon comes from the exe's embedded resource.

### Trigger & idempotence

- On GUI startup, after the window is up, a **background thread** runs a
  cheap staleness check: a version marker (CFBundleShortVersionString in the
  plist / `X-Crew-Version` key in the .desktop file / a sidecar marker for the
  .lnk) is compared to `CARGO_PKG_VERSION`, and the symlink/Exec target is
  compared to the current binary path. Only on mismatch does it rewrite.
  Nothing runs on the winit thread.
- `crew install-app` runs the same registration synchronously and prints what
  it wrote. `crew install-app --remove` deletes the artifacts.
- `install.sh` invokes `crew install-app` after installing the binary
  (best-effort; failure doesn't fail the install).
- `CREW_NO_APP_INSTALL=1` disables the automatic path entirely.

## 4. Runtime icon fixes

Registration covers the menus; these cover the live process regardless of how
it was launched (Terminal launches especially):

- **macOS Dock:** at startup set `NSApplication.applicationIconImage` from the
  embedded 512 px PNG, using the `objc2`/`objc2-app-kit` crates winit already
  depends on. This corrects the Dock icon even when the process isn't
  bundle-launched (symlinked bundles can lose bundle identity; terminal
  launches never had it).
- **Windows / Linux:** `winit` `Window::set_window_icon` from the embedded
  PNG, and set the Wayland app_id / X11 WM_CLASS to `crew` so compositors
  match the window to the `.desktop` entry (`StartupWMClass=crew`).

## 5. Testing

- Unit tests for plist and .desktop generation (golden strings), path
  resolution, and idempotence: register twice → second run writes nothing;
  bump version marker → rewrite happens.
- Registration functions take a base-dir parameter so tests write into a
  tempdir instead of the real `$HOME`.
- macOS end-to-end via the existing GUI verify harness (isolated `$HOME`):
  run `crew install-app`, assert bundle structure, `open -a` smoke check,
  screenshot to confirm the Dock icon.

## Out of scope (YAGNI)

Code signing / notarization, .dmg or MSI installers, file-type associations,
"Open at login", Launchpad pinning, taskbar pinning.
